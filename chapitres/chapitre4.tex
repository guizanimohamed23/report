\section{Introduction}

\indent\justify This sprint focuses on the scanning pipeline and on the vulnerability processing features of the HexStrike platform. The goal of this iteration is to implement reliable scan orchestration, normalize heterogeneous tool outputs into a unified schema, match findings against the vulnerability catalog, and persist results for later analysis and reporting.

\indent In practice, this sprint covers scan submission, queueing and scheduling, worker orchestration, normalization, vulnerability matching, result persistence, and the user interfaces to submit and view scans.

\subsection{Sprint Purpose}

\indent\justify The purpose of this sprint is to build a reliable and scalable scanning infrastructure that transforms raw target submissions into actionable vulnerability intelligence. Concretely, the sprint delivers a robust queue and worker system for tool orchestration, a normalization pipeline that unifies outputs across heterogeneous security tools, a matching engine that correlates findings with known vulnerabilities, and a user interface for scan management and result analysis. This foundation enables efficient vulnerability detection and establishes the data pipeline required for advanced attack graph generation in subsequent sprints.

    \small

        \hline
        \textbf{Name} & Submit Scan Target \\\hline

        \textbf{Description} & This use case enables an authenticated user to initiate a security scan by providing a target URL and optional parameters. The system validates the input, creates a Scan job record, enqueues it for processing, and returns a unique job identifier to the user for tracking purposes. \\\hline
        \textbf{Precondition} & The user is authenticated with a valid session token. The scan submission endpoint is operational and the queue service is accessible. Network connectivity to target validation services is available. \\\hline
        \textbf{Nominal Scenario} & 1. User navigates to the Scanner page. 2. User enters a target URL and optionally selects scan parameters (tools, depth, timeout). 3. User initiates the scan submission. 4. System validates the URL format and accessibility. 5. System checks target against blacklist/whitelist policies. 6. System creates a Scan record in the database with metadata. 7. System enqueues the job to the scan queue. 8. System generates a unique job identifier. 9. System returns the job ID and redirect user to scan history or progress page. \\\hline
        \textbf{Exception Scenario} & 1. Invalid URL format: System displays validation error and prompts for correction. 2. Target unreachable: System validates after submission and notifies user of failure. 3. Target blacklisted: System rejects the submission with a policy violation message. 4. Queue service unavailable: System logs the error and displays a retry message. 5. Database error: System logs error and displays a generic failure message. 6. Rate limit exceeded: System informs user that too many scans are queued and suggests waiting. \\\hline
        \textbf{Postcondition} & A Scan job is created in the database with status ``pending'', the job is enqueued for worker processing, and the user receives a job identifier for tracking. \\\hline
        \textbf{Logical Constraints} & URLs must be validated for format and accessibility. Blacklist/whitelist policies must be enforced. Rate limiting must be applied per user. Job identifiers must be globally unique. Queue must maintain FIFO ordering. \\\hline
    \end{tabularx}
\end{table}
