\section{Introduction}

This sprint is dedicated to attack graph generation. The goal is to transform project artifacts into diagrams that show how data, endpoints and vulnerabilities are connected, so the report can highlight potential attack paths in the platform.

\subsection{Objectives}

The chapter aims to:
\begin{itemize}
  \item Describe the graph generation pipeline and the diagram types used in this sprint.
  \item Show how backend artifacts such as the Prisma schema and API routes are converted into diagrams.
  \item Explain how the generated graph assets are exported and embedded in the report.
  \item Present the implementation of the attack graph generation workflow.
\end{itemize}

\section{Sprint Purpose}

The purpose of this sprint is to generate a clear visual representation of the backend structure and of the attack surface exposed by the application. Based on the backend documentation, the Prisma schema and the API layers, the sprint produces diagrams that help connect entities, requests and vulnerabilities in a single view.

\section{Requirements Analysis}

\subsection{Generation Pipeline}

The generation pipeline implemented in the project follows three main steps:
\begin{enumerate}
  \item Extraction: parse source artifacts such as \texttt{pfe/PFE\_BACKEND/backend/prisma/schema.prisma}, backend route definitions and supporting documentation.
  \item Transformation: map extracted entities to an intermediate representation of nodes and edges; enrich nodes with vulnerability metadata when available.
  \item Rendering: emit diagram source with Mermaid or diagrams.net and export the final asset for the report.
\end{enumerate}

\section{Implementation}

This sprint is implemented with a pipeline that reads the backend artifacts, converts them into a graph model and exports the final diagram assets.

The backend README describes the pipeline as a normalization and intelligence engine. For graph generation, the relevant inputs are the Prisma schema, API routes and the security metadata extracted from the backend. The backend schema provides the entity relationships, while the API layer and generated evidence help connect endpoints to vulnerabilities.

The implementation relies on Mermaid-based diagram drafting for quick iteration and on report assets stored under \texttt{uploads/diagrams/}. The generated diagrams are then referenced from the LaTeX report using labeled placeholders until the final assets are inserted.
